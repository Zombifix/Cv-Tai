Le mode “Smart Import Experience” ne gère pas correctement les formats structurés.

Problème observé :
- Quand je colle un JSON complet de profil contenant `profile`, `experiences`, `global_skills`, les champs se remplissent mal.
- Exemple : `title` devient `{`, `company` devient `Unknown`, `summary` contient une partie du JSON brut, puis la sauvegarde échoue.
- Le composant semble traiter tout input comme du texte libre à parser, alors qu’il faut supporter aussi des JSON structurés.

Je veux corriger ça proprement.

Objectif :
Ajouter une vraie détection de format dans le Smart Import d’expérience, avec 3 cas supportés :
1. texte libre décrivant une seule expérience
2. JSON décrivant une seule expérience
3. JSON complet contenant `profile` + `experiences[]`

Comportement attendu :
- Si l’input est du texte libre : utiliser le parsing actuel / IA
- Si l’input est un JSON valide avec une seule expérience :
  - mapper explicitement les clés :
    - `role` -> `title`
    - `company` -> `company`
    - `start` -> `startDate`
    - `end` -> `endDate`
    - `bullets[]` -> summary ou description concaténée
    - `skills[]` -> tags si supportés
- Si l’input est un JSON contenant `experiences[]` :
  - ne pas essayer de le mettre directement dans la modale d’une seule expérience
  - afficher une étape intermédiaire de sélection :
    - liste des expériences détectées
    - possibilité d’en choisir une à importer
    - ou bouton “Import all”
- Si `end = "Present"`, convertir en :
  - `isCurrent = true`
  - `endDate = null`
  ou autre format attendu par l’app
- Si parsing impossible :
  - afficher une erreur claire
  - garder le texte brut dans la zone d’import
  - ne pas remplir les champs avec des valeurs absurdes comme `{`

Je veux aussi :
- une fonction de mapping dédiée, pas seulement un prompt LLM
- une validation avant save
- une normalisation des dates (`YYYY-MM`, `YYYY-MM-DD`, `Present`)
- un fallback propre si certaines données sont absentes

Merci de proposer :
1. la logique de détection de format
2. la fonction de transformation vers le modèle interne de l’expérience
3. l’UX de sélection si plusieurs expériences sont détectées
4. la correction de la sauvegarde
